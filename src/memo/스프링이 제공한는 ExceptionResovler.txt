스프링 부트가 기본으로 제공하는 ExceptionResolver 는 다음과 같다.
HandlerExceptionResolverComposite 에 다음 순서로 등록
1. ExceptionHandlerExceptionResolver(대부분 이걸로 API 오류 응답 처리, 가장 중요해서 뒷쪽에서 자세히 학습하는 걸로 하자)
2. ResponseStatusExceptionResolver(@ResponseStatus 애노테이션을 처리)
3. DefaultHandlerExceptionResolver
-> 2,3번은 잘 사용하지않아 그렇게 중요하지 않으니 먼저 2,3를 빠르게 이번 시간에는 학습한다.



- ResponseStatusExceptionResovler -

@ResponseStatus(code= HttpStatus.BAD_REQUEST,reason = "잘못된 요청 오류")
public class BadRequestException extends RuntimeException{
}
-> @ResponseStatus를 보고 ResponseStatusExceptionResolver가 선택이 된다.
여기에서의 핵십은 RuntimException에러의 httpstatus 500을 Resovler에서 가로채서 response.sendError에서 우리가 설정한 HttpStatus.BAD_REQUEST로 예외를 바꿔치기 한다.
그러나 이건 WAS에서 다시 재요청을 발생시킨다.


@ResponseStatus는 수정할 수 없는 외부 예외라던지 그런 곳에서는 사용이 불가능하다.
그럴 떄에는 다이렉트로 아래와 같이 throw new ResponseStatusException을 사용하면 된다. 이것도 ResponseStatusExcpetionResolver가 처리해준다.
ex)
@GetMapping("/api/response-status-ex2")
public String responseStatusEx2() {
    throw new ResponseStatusException(HttpStatus.BAD_REQUEST,"error_bad",new IllegalArgumentException());
}


- DefaultHandlerExceptionResolver -
이건 스프링 【내부】에서 발생하는 예외를 처리한다.
예를 들어 Binding 중 타입이 맞지 않는 것이 들어 오는 경우 스프링은 TypeDisMatch 예외를 터트린다.
그러나 이건 클라이언트의 잘못이기 때문에 500 대신에 4xx 에러를 터트려야 한다.(만약 이걸 처리하는 리졸버가 없으면 WAS에 그대로 500에러가 발생된다)
그래서 이러한 스프링 내부 오류의 경우 DefaultHandlerExcpetionResolver가 TypeDisMatch 예외(스프링 내부 오류)를 4xx로 변환을 해준다,
그러나 이것 또한 역시 response.sendError를 사용하기에 WAS가 재요청을 한다.


- ExceptionHandlerExceptionResolver(@ExcpetionHandler) -
스프링이 제공하는 ExceptionResolver를 사용하기 이전에는 우리가 직접 ExceptionResovler를 구현을 하였다.
그러나 이는 매우 복잡하고, 또한 ResponseStatusExcpetionResolver, DefaultHandlerExceptionResolver를 포함하여 이러한 것들은 response.sendError를 사용하기 때문에
스프링 부트가 제공하는 BasicErrorController가 제공하는 API 응답 오류 양식을 따른다.
이 과정에서 항상 WAS가 재요청을 하는 등 과정이 복잡하였다. 우리는 이를 해결하기 위하여 UserHandelrExcpetionHandler를 직접 구현하였다.
response.getWriter()를 사용하여 해결하여서 재요청 문제는 해결하였지만 return new ModelAndVIew를 강제적으로 반환해야 했다. API 오류 응답 에러인데 return new ModelaAndView
를 반환하는 것은 뭔가 이상하다.
그리고 API 오류 응답의 형태는 매우 다양하다. 예를 들어 회원 컨트롤러와 주문 컨트롤러에서 발생하는 API 오류 응답을 각각 다르게 만드는 것은 기존 방식으로는 매우 어렵다. 만약 한다고 하면은
어떤 컨트롤러에서 왔는지 if 문등을 덕지덕지 발라야 한다.
그래서 우리가 기존에 배운 방식의 ExceptionResolver로는 한계가 있다.
ExceptionHandlerExceptionResolver는 이러한 문제점을 해결해준다
(참고로 오류 페이지 화면 같은 경우에는 스프링부트의 BasicErrorController의 방식을 사용하는 것이 좋다. 왜냐하면 오류 화면 페이지는 다양할 필요가 없기 때문이다)
