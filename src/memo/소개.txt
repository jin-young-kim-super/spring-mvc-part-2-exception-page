예외 처리를 이해하려면 
STEP 1. 서블릿이(서블릿 컨테이너)이 어떻게 예외 처리를 하는지
STEP 2. 서블릿 컨테이너 위에서 터진 예외를 스프링이 어떻게 쉽게 처리해주는 지
-> 이 두 단계를 이해해야지만 예외 처리를 이해할 수가 있으므로 STEP1과 STEP2를 숙지 해야 한다 반드시!



서블릿의 오류 처리 방식은 크게 2가지이다.
1. Exception(WAS가 "아! 뭔가 런타임 중 서버【문제】가 발생했구나")
- 예외가 발생하여 WAS까지 예외 객체가 전달되는 경우(개발자가 직접 [throw 예외]하는 것도 해당)
- 예: NullPointerException, RuntimeException

예외를 잡지 못하는 경우의 예외의 흐름
WAS(여기까지 전파) <- 필터 <- 서블릿 <- 인터셉터 <- 컨트롤러(예외발생)
-> 결국 톰캣 같은 예외가 서블릿(서블릿 컨테이너)를 넘어서 was까지 전달이 된다.
그럼 WAS는 이 예외를 어떻게 처리할 까?
만약 WAS는 자기한테까지 예외 객체가 도착을 해버리면 "어? 서버에서 못 잡은 예외가 터졌구나"하고 무조~~~~건 5xx 에러 페이지를 만든다.


2. response.sendError()("아! 이건 【문제】가 아니라 개발자의 정상 요청이구나")
- 개발자가 HTTP 오류 상태 코드를 직접 지정하는 경우
- 예: 404 Not Found, 400 Bad Request
- 중요 : 예외 객체를 던지는 것이 아니라 오류 응답을 요청하는 것(WAS는 그 오류 응답에 맞춰서 에러 페이지를 만든다)
-> 예외 객체를 WAS에 던지는 것이 아니라 was에게 오류 처리를 요청하는 것!

sendError() 실행 흐름
WAS(sendError 호출 기록 확인) <- 필터 <- 서블릿 <- 인터셉터 <- 컨트롤러
-> WAS는 이것을 서버 에러에 의한 예외라고 인식하는 것이 아니라 그냥 개발자가 sendError에 담은 오류 정보대로 에러 페이지를 만들어 줄 뿐이다.

이번 시간에 각가의 에러 페이지를 보았다. 그러나 이 에러 페이지 화면은 사용자들에게는 매우 불친절하다
-> 다음 시간에는 에러 페이지를 사용자 친화적으로 만드는 방법을 학습하겠다.
